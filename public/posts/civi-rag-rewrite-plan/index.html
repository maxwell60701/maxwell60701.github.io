civi-rag.md 改写计划#

一、当前文档问题分析#

  1. 结构混乱：数据采集代码和 RAG 服务代码混在一起，逻辑不清晰
  2. 代码冗余：直接粘贴大量原始代码，缺少注释和分层
  3. 缺失系统架构图：没有整体流程图
  4. 缺少数据字典：15 个 collection 没有说明
  5. 部署说明简陋：只有技术栈列表，没有 Docker 部署细节
  6. 设计思路不明确：路由判定、多路检索等核心设计没有深入解释

二、新文档结构规划#

1. 项目概述
   - 目标：基于文明6构建 RAG 问答系统
   - 核心能力：支持多主题联合问答

2. 系统架构
   - 架构流程图（文字版 Mermaid）
   - 组件说明：数据采集 → 向量化 → 路由检索 → 生成

3. 数据层
   - 15 个 Milvus Collection 详解
   - 数据来源：Civilopedia 网站
   - 文档拆分策略

4. 核心模块
   4.1 数据采集模块
       - 爬虫策略
       - 解析逻辑

   4.2 向量化存储
       - Embedding 模型
       - 批量入库流程

   4.3 路由判定
       - LLM 判断问题涉及的表
       - 结构化输出保证 JSON 稳定

   4.4 多路检索与生成
       - 并行检索策略
       - 聚合上下文生成

5. API 接口
   - 接口定义
   - 请求/响应示例

6. 部署指南
   - Docker Compose 编排
   - 环境变量配置
   - 服务访问地址

7. 关键设计思路
   - 提示词设计原则
   - 为什么做路由判定
   - 多路检索的优势

三、内容优化要点#

3.1 替换冗余代码为分层架构图#

当前问题：原始代码 200+ 行直接堆砌

优化方案：

  • 用 Mermaid 流程图展示数据流
  • 代码拆分为 4 个模块：采集器、存储引擎、路由服务、生成服务
  • 每个模块保留核心逻辑（20 行以内），配详细注释

3.3 重新设计提示词章节#

拆分当前混杂的提示词为：

  • 路由判定提示词：判断问题涉及哪些表
  • 生成回答提示词：严格按 source 逐条列出

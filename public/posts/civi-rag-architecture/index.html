文明6 RAG 知识库 - 项目架构说明#

一、项目概述#

本项目基于《文明6》Civilopedia 构建 RAG（检索增强生成）问答系统，支持用户询问游戏相关的各类问题（如伟人、建筑、奇观、兵种等），系统通过路由判定自动选择相关知识库进行检索，聚合后生成回答。

数据来源：Civilopedia（文明6百科）

三、数据层设计#

3.1 知识库划分#

共 15 个独立的 Milvus Collection，按主题分类：

Collection 说明 示例内容
belief 宗教信仰 信仰名称、效果描述
religion 宗教体系 宗教类型、相关领袖
country 文明国家 文明特性、起始单位
leader 领袖 领袖名称、特色能力
building 建筑 建筑名称、成本、效果
wonder 奇观 奇观名称、建造条件、产出
resource 资源 资源类型、分布区域
citystate 城邦 城邦名称、增益类型
district 城区 城区名称、建造条件
feature 地形地貌 地形效果、改良适配
great_person 伟人 伟人名称、所属时代、特长
improvement 改良设施 改良名称、建造条件、产出
moment 历史时刻 时刻名称、触发条件、奖励
unit 单位 单位名称、强度、升级路径
unit_promotion 单位晋升 晋升名称、效果、前置条件

四、核心模块设计#

4.1 路由判定模块#

设计目的：用户问题可能涉及多个领域（如"秦始皇的特色单位是什么？“涉及 leader 和 unit），需要先判定问题涉及哪些知识表。

工作流程：

用户问题 → LLM 分类判定 → 结构化输出 → 确定检索范围

判定策略：

  1. 使用 LLM（gpt-oss:120b-cloud）分析问题，输出自然语言描述的判断结果
  2. 二次调用结构化模型（llama3.1）配合 with_structured_output(RouteQuery) 保证 JSON 稳定可解析
  3. RouteQuery 定义了 15 种表的字面量类型，LLM 输出必须在指定范围内

RouteQuery 结构：

class RouteQuery(BaseModel):
    tables: List[Literal[
        "belief", "religion", "country", "leader", "building", "wonder",
        "resource", "citystate", "district", "feature", "great_person",
        "improvement", "moment", "unit", "unit_promotion"
    ]] = Field(description="这个问题与哪些关键字有关，返回对应的表名")

4.2 多路检索模块#

设计目的：支持跨多个知识表的联合查询，需要并行检索后聚合。

工作流程：

判定结果 tables=[A, B, C]
       │
       ▼
┌─────────────────────────────────────────────┐
│  为每个表创建独立检索器                       │
│  retriever_dict = {                         │
│      "leader": vectorstore.as_retriever(),  │
│      "unit": vectorstore.as_retriever(),    │
│      ...                                    │
│  }                                          │
└─────────────────────────┬───────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────┐
│  RunnableParallel 并行执行                    │
│  - 各检索器同时执行，互不阻塞                 │
│  - 结果自动聚合为 context                    │
└─────────────────────────┬───────────────────┘
                          │
                          ▼
                    聚合后的上下文

4.3 生成回答模块#

设计原则：

  1. 严格基于上下文：禁止使用外部知识
  2. 按 source 逐条列出：不合并不同文档的内容
  3. 禁止省略：有多少条就列多少条
  4. 禁止推断：只陈述事实，不做推测

RAG Chain 构造：

RunnableParallel({
    "context": retriever,        # 检索到的上下文
    "question": RunnablePassthrough()  # 原始问题
}) | prompt | llm | StrOutputParser

五、服务接口#

5.1 FastAPI 服务#

端点：POST /ask

请求：

{
    "question": "列出所有科学类伟人"
}

响应：

{
    "answer": "科学类伟人包括：\n\n阿尔伯特·爱因斯坦（source: ...）：...\n居里夫人（source: ...）：...\n..."
}

5.2 API 路由流程#

POST /ask
    │
    ▼
generate_answer(question)
    │
    ├── 路由判定 → 确定涉及哪些 Collection
    ├── 多路检索 → 并行获取上下文
    └── 生成回答 → LLM 整合输出
    │
    ▼
AskResponse(answer=...)

七、关键技术选型#

组件 选型 原因
LLM（路由） gpt-oss:120b-cloud 云端大模型，推理能力强
LLM（生成） llama3.1 本地部署，可控性强
Embedding embeddinggemma 高效的向量编码
向量数据库 Milvus v2.6.4 支持分布式、高可用
链路框架 LangChain 丰富的 RAG 组件支持
API 框架 FastAPI 高性能、自动文档

八、设计亮点#

  1. 路由判定：避免全量检索，节省资源和时间
  2. 结构化输出：二次调用确保 JSON 稳定性
  3. 多路并行：RunnableParallel 实现真正的并行检索
  4. 严格回答：提示词约束禁止省略和推断，保证答案准确性
  5. 按需扩展：新增知识领域只需新增 Collection 和采集器
